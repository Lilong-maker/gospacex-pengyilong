# 订单超时自动关单时序图说明

本文档解释 `sequence.puml` 中的时序图流程。

---

## 流程概览

整个流程分为 **4个阶段**:

```
+----------------+    +----------------+    +----------------+    +----------------+
|   ① 订单创建   | -> |   ② 超时检查   | -> |   ③ 自动关单   | -> |   ④ 库存回滚   |
+----------------+    +----------------+    +----------------+    +----------------+
```

---

## 详细步骤说明

### 阶段一: 订单创建 (步骤 ①-⑥)

用户下单后，系统需要完成以下操作:

| 步骤 | 操作 | 说明 |
|------|------|------|
| **①** | 提交订单 | 用户点击"提交订单"按钮 |
| **②** | 锁定库存 | 系统预扣商品库存，防止超卖 |
| **③** | 保存订单 | 订单写入数据库，状态为 **PENDING(待支付)** |
| **④** | 设置过期标记 | 在Redis中设置30分钟过期时间 |
| **⑤** | 注册超时任务 | 调度服务将在30分钟后检查此订单 |
| **⑥** | 返回结果 | 告诉用户订单创建成功，需要在30分钟内支付 |

**关键点:**
- 此时订单状态 = `PENDING` (待支付)
- 库存已被锁定，其他用户无法购买这部分库存
- 系统开始倒计时30分钟

---

### 阶段二: 超时检查 (步骤 ⑦-⑧)

30分钟后，系统自动触发检查:

| 步骤 | 操作 | 说明 |
|------|------|------|
| **⑦** | 触发检查 | 调度服务时间到了，询问订单服务 |
| **⑧** | 查询状态 | 从数据库读取订单当前状态 |

**分支判断:**
```
如果状态 = PAID(已支付) 
    └── 什么也不用做，流程结束
    
如果状态 = PENDING(未支付)
    └── 继续执行自动关单流程
```

---

### 阶段三: 自动关单与库存回滚 (步骤 ⑨-⑬)

订单超时未支付，系统自动取消:

| 步骤 | 操作 | 说明 |
|------|------|------|
| **⑨** | 执行取消 | 调度服务通知订单服务取消订单 |
| **⑩** | 更新状态 | 订单状态改为 **CANCELLED(已取消)** |
| **⑪** | 释放库存 | 将锁定的库存退回到可用库存池 |
| **⑫** | 清理缓存 | 删除Redis中的过期标记 |
| **⑬** | 完成任务 | 告诉调度服务任务已完成 |

**关键点:**
- 订单正式关闭，用户无法再支付
- 库存已释放，其他用户可以购买
- 系统资源已清理

---

### 替代流程: 用户成功支付 (步骤 ⑭-⑱)

如果在30分钟内用户完成支付:

| 步骤 | 操作 | 说明 |
|------|------|------|
| **⑭** | 用户支付 | 用户完成付款操作 |
| **⑮** | 更新状态 | 订单状态改为 **PAID(已支付)** |
| **⑯** | 清理缓存 | 删除Redis中的过期标记 |
| **⑰** | 取消定时任务 | 告诉调度服务:"不用检查这个订单了" |
| **⑱** | 返回成功 | 告诉用户支付成功 |

**关键点:**
- 订单状态变为 `PAID`，不会被自动取消
- 超时检查任务被取消，不会执行关单操作
- 库存保持锁定状态，等待后续发货

---

## 时序图颜色说明

时序图中使用不同颜色区分不同类型的操作:

- **浅蓝色 (LightBlue)**: 订单服务相关操作
- **浅绿色 (LightGreen)**: 库存服务相关操作  
- **浅黄色 (LightYellow)**: 调度/定时任务相关
- **粉红色 (Pink)**: 数据库操作
- **浅珊瑚色 (LightCoral)**: 取消/回滚操作

## 常见问题

### Q1: 为什么要先锁定库存？
**A**: 防止超卖。如果不下单就锁库存，可能出现100个人同时买最后1件商品的情况。

### Q2: 30分钟是怎么实现的？
**A**: 使用延迟队列(Delay Queue)。订单创建时向队列投递一个"30分钟后执行"的任务，时间到了自动触发检查。

### Q3: 如果库存回滚失败了怎么办？
**A**: 系统会记录失败日志，进入重试队列，最多重试3次。如果仍然失败，会发送告警通知人工处理。

### Q4: 用户支付和自动关单同时发生会怎样？
**A**: 使用数据库状态机保证幂等性。只有状态为 `PENDING` 的订单才能被取消，已支付的订单不会重复取消。

### Q5: Redis在这里起什么作用？
**A**: 两个作用:
1. **快速判断**: 检查订单是否存在过期标记，比查数据库快
2. **兜底保护**: 即使调度服务出问题，Redis过期事件也能触发关单

## 数据状态流转

```
订单创建
    |
    v
+---------+
| PENDING | <-------- 待支付状态
| (待支付) |
+----+----+
     |
     +--------------+
     |              |
     v              v
+---------+   +-----------+
|  PAID   |   | CANCELLED |
| (已支付) |   | (已取消)   |
+---------+   +-----------+
     |              |
     v              v
  等待发货      订单关闭
```
